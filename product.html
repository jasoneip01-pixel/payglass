1. 产品定位

1.1 是什么

PayGlass 是跨协议 Agent 支付可观测性引擎。它不在支付流中（不处理资金），而是在支付流之上（观测、追踪、诊断）。

Agent 支付栈
┌──────────────────────────────────────┐
│            PayGlass                   │  ← 我们在这里：观测层
│   Dashboard · Tracer · Observers      │
├──────────────────────────────────────┤
│  Stripe SPT  │  x402  │  Circle Nano │  ← 协议层
├──────────────────────────────────────┤
│  Card/ACH    │  USDC  │  USDC        │  ← 结算层
└──────────────────────────────────────┘

1.2 不是什么

  • ❌ 不是支付协议——不处理资金流
  • ❌ 不是支付网关——不托管私钥
  • ❌ 不是合规工具——不需要牌照
  • ❌ 不是 Stripe/x402/Circle 的竞争对手——是它们的观测层
核心隐喻：Datadog 让 DevOps 在一面玻璃上看所有服务的健康状态。PayGlass 让 Agent 开发者在一面玻璃上看所有支付的状态。
⚠️ 2026 年 5 月最新研究指出：
x402 类 Agent 支付存在授权绑定不足、重放攻击、并发竞态、动态定价超标等安全风险
（arXiv:2605.11781 · arXiv:2605.30998）
这些风险 Etherscan 看不见，普通 HTTP 日志也看不见。PayGlass 的 trace 设计本身就是安全审计层。

2.2 开发者的五大痛点

痛点现状PayGlass 解法
协议选型瘫痪3 个协议，各有优劣，不知道选哪个三轨统一 Dashboard，同屏对比延迟/成本/错误率
支付流黑箱EIP-3009 签名→TEE 验证→批量结算，6+ 步中间态，任何一步失败都是黑箱逐事件追踪，可视化管线，精确定位失败点
跨链不可见11 条链+3 个网关，一笔支付跨链后消失在黑洞里跨链追踪，batch→tx→confirmation 全链路可见
支付安全盲区重放攻击、重复支付、paid-but-denied、allowance 超标——Etherscan 看不到，日志也看不到内置安全检测引擎，自动标记 replay/duplicate/anomaly
协议性能无从对比x402 delay vs Circle Nano delay vs Stripe MPP delay——没有客观数据跨协议 benchmark，延迟/失败率/结算时长实时对比

4. 竞争格局

观测范围协议覆盖资金处理牌照需求
PayGlass全部x402 + Nano + SPT❌ 不碰❌ 无

Etherscan链上交易全部链上❌ 不碰❌ 无
PayGlass 的差异化：Etherscan 能看到链上结算，但看不到链下的签名→验证→批量过程。PayGlass 覆盖的是 EIP-3009 签名到链上结算之间的灰色地带——这是 Etherscan 盲区，也是开发者最痛的调试区。

4.2 威胁与应对

威胁概率影响应对
Stripe 把观测做到 MPP 里中高不被锁定在 Stripe 生态——跨协议是护城河
Circle Gateway 开源观测模块低中单协议观测≠跨协议，碎片化越严重我们越有价值
AWS AgentCore 自带监控中中云厂商监控是基础设施级，不是应用级——支付流可视化更上层

5.2 核心数据模型

TraceEvent {
  trace_id:      唯一追踪 ID
  protocol:      x402 | circle-nanopayments | stripe-spt
  event_type:    signature | verification | batch | settlement
  status:        pending → verified → batched → settled
  amount_usdc:   金额（统一 USDC 计价）
  source_agent:  付款 Agent 标识
  target_service:收款方/资源 URL
  chain:         区块链
  tx_hash:       交易哈希
  batch_id:      批次 ID
  risk_flags:    [replay? | duplicate? | allowance_overdraft? | binding_mismatch?]
}

5.3 标准仪表化（OpenTelemetry 兼容）

PayGlass 不只是 Dashboard——它是标准化的支付仪表化层：

payglass.trace()        # 创建支付追踪
payglass.middleware()   # 自动截获 HTTP/Webhook
payglass.collector()    # 汇聚 TraceEvent
payglass.exporter()     # 导出至 OTEL/自有 Dashboard
payglass.dashboard()    # Web UI
设计原则：兼容 OpenTelemetry，进入现有 DevOps 栈。这工具不是新习惯，而是已有观测体系的 payment extension。

7. 安全风控：不只是看得见，而是识别危险

核心洞察——这才是 PayGlass 的真正付费点：
免费用户看 Dashboard，付费用户买 Risk Alerts / Security Audit / Anomaly Detection。
支付的观测数据就是安全审计数据——同一份数据，两个价格点。

2026 年 5 月两项独立研究揭示了 x402 类 Agent 支付的系统性安全风险。PayGlass 的观测层天然具备捕捉这些风险的能力。

7.1 PayGlass 能检测的风险

风险类型描述检测方法严重度
🔁 重放攻击同一签名被多次提交nonce 重复检测高
📋 重复支付同一支付意图触发多次结算trace_id 唯一性验证高
⛔ Paid-but-Denied签名已确认但资源未交付签名事件 → 无交付事件检测高
👻 Unpaid-but-Served资源已交付但无对应付款资源交付 → 无对应签名事件检测高
🔗 绑定不匹配签名绑定的资源与交付的资源不一致EIP-3009 request-binding 字段验证高
🧊 批量结算卡住Batch pending 超过阈值batch_id 到 tx_hash 时间窗口监控中
💰 Allowance 超标动态定价导致实际扣减超出授权额度allowance override 检测中
📊 并发竞态同一资源被多笔支付同时请求target_service 并发计数器低

7.2 为什么别人做不到

工具为什么看不到
Etherscan只看链上 tx_hash，看不到链下签名→验证→批量过程
Stripe Dashboard只看 Stripe 内部，看不到 x402 / Circle Nano
HTTP 日志只记录请求/响应，看不到跨协议的完整状态机
Prometheus + Grafana只能自定义监控指标，无法追踪单笔支付语义
网络效应壁垒：PayGlass 沉淀跨协议性能基准数据——x402 在 Base 的结算耗时分布、Circle Nano 在 11 条链的稳定性排名、哪些 Gateway 重复失败率高、哪些 Agent 消费行为异常。这就是从工具到 intelligence layer 的路径。

10. 风险与应对

风险等级应对
窗口关闭（Stripe/Circle 统一底层）高12-18 个月内建立开发者心智——"调试 Agent 支付 = PayGlass"
市场太小（Agent 支付还没规模）中免费层 + 社区驱动，等到 2027 年第一批付费用户
被巨头"插件化"高不做单一协议的 Dashboard。 通过开源 SDK + OTEL 兼容 + 自建 Collector 保持独立性。
Veridex 加 Dashboard中先做安全层——Veridex 协议抽象 + Dashboard ≠ 安全审计 + 异常检测
x402 被边缘化中协议无关设计——新协议出现，加一个 Observer 即可
合规要求变化低不处理资金 = 不受支付牌照约束

10. 产品原则

  1. 不选边。 PayGlass 不给任何协议背书。观测一切，推荐零。
  2. 不可见即不存在。 支付流中的每一个中间态都必须可见。
  3. 开发者优先。 产品体验从 CLI 开始，不是从销售演示开始。
  4. 免费层必须真的好用。 如果免费层是残废的，品类教育不会发生。
  5. 不碰资金。 这是护城河，也是安全边界。
